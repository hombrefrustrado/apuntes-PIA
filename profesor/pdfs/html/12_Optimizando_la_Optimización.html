Cuando empezamos a hablar del ajuste de los par√°metros, establecimos que esencialmente el algoritmo de ajuste que usamos es el Descenso del Gradiente Estoc√°stico (Stochastic Gradient Descent o SGD) y que en Pytorch lo tenemos como el optimizador torch.optim.SGD. A priori, SGD utiliza una regla de ajuste sencilla para los par√°metros del modelo.

$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \nabla L(m_{\mathbf{w}_{t}}(x))$$

que se descompone para cada par√°metro:

$$w_{t+1,i} = w_{t,i} - \gamma\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}}$$

siendo $\mathbf{w}_{t} = (w_{t,1},...,w_{t,n})$ los par√°metros del modelo $m$ en el psao de entrenamiento $t$ y $\gamma$ el ratio de aprendizaje. Simplificamos la nomenclatura de la funci√≥n de p√©rdida omitiendo la salida conocida.

Siendo m√°s exactos, a priori esta es la regla de aprendizaje para el descenso del gradiente si se usan todos los datos para el c√°lculo de las derivadas parciales. De esa manera obtenemos un √∫nico gradiente promediado para actualiuzar el error. Si usamos un solo dato elegido aleatoriamente, hablamos del descenso del gradiente estoc√°stico y, cuando usamos un lote de datos nos podemos referir al m√©todo como descenso del gradiente por lotes. Conceptualmente es muy similar, pero como hemos comentado en otros cuadernos, aumentar el n√∫meros de datos sobre los que calcular el gradiente afecta la cantidad de ruido que tiene dicho gradiente.

Vamos a recapitular algunos conceptos:

Un gradiente calculado con todos los datos tiene el m√≠nimo ruido posible, pero su camino se vuelve determinista (no hay ning√∫n elemento estoc√°stico que incluya variabilidad) y si en ese camino encuentra un m√≠nimo local dif√≠cilmente va a poder salir de √©l. Un gradiente calculado con un solo ejemplo va a tener mucho ruido porque est√° a merced de lo bien que se ajuste un √∫nico ejemplo al modelo. Esto a√±ade un indeterminismo √∫til para salir de m√≠nimos locales (lo que es un m√≠nimo local para todos los datos en su conjunto, puede no serlo para un √∫nico dato), pero hace el entrenamiento m√°s inestable y dificulta la convergencia (adem√°s de no aprovechar la paralelizaci√≥n en GPU).

El aprendizaje profundo entrena por lotes, que es un t√©rmino intermedio, m√°s cerca de uno u otro extremo seg√∫n el tama√±o de los lotes de entrenamiento en relaci√≥n al total de datos. Aunque en una √©poca recorramos todo el conjunto de datos, el ajuste lo hacemos en cada paso (con cada lote). El gradiente de las ecuaciones es, por tanto, la media de los gradientes para cada dato en el lote.

La cuesti√≥n ahora es: ¬øpodemos mejorar esa manera tan b√°sica de ajustar nuestro modelo para que el entrenamiento sea m√°s r√°pido, estable y evite caer en m√≠nimos locales? Hay algunas t√©cnicas comunes que vamos a ver, pero antes vamos a recuperar algo de c√≥digo de otros cuadernos para los ejemplos.

class CudaDataset(Dataset):
  def __init__(self, dataset, device, transform = None):
    self.dataset = dataset
    self.cuda_y = []
    self.cuda_x = []
    self.device = device
    self.transform = transform

    for x, y in tqdm(self.dataset, desc="Moving to GPU"):
      self.cuda_x.append(x.to(self.device))
      self.cuda_y.append(torch.tensor(y, device=self.device))

  def __len__(self):
    return len(self.dataset)

  def __getitem__(self, idx):
    if self.transform is None:
      x = self.cuda_x[idx]
    else:
      x = self.transform(self.cuda_x[idx])
    return x, self.cuda_y[idx]

def split_dataset(dataset, split_share=0.5):
  """
  Devuelve dos subconjuntos del dataset. split_share define cu√°ntos ejemplos ir√°n al
  primer subconjunto. El resto ir√°n al segundo.
  """
  mask_indices_to_first_subset = torch.rand(len(dataset))<=split_share
  indices_first_subset = [i for i, (_, _) in enumerate(dataset) if mask_indices_to_first_subset[i]]
  indices_second_subset = [i for i, (_, _) in enumerate(dataset) if not mask_indices_to_first_subset[i]]

  first_subset = torch.utils.data.Subset(dataset, indices_first_subset)
  second_subset = torch.utils.data.Subset(dataset, indices_second_subset)

  return first_subset, second_subset

def learning_loop_for_classification(train_dataloader, val_dataloader, model, epochs, loss_fn, optimizer, validation_freq):
  epoch_loss_list = []
  val_loss_list = []
  val_acc_list = []

  with tqdm(range(epochs), desc="epoch") as pbar:
    for epoch in pbar:
      steps_loss_list = []
      for x_true, y_true in train_dataloader:
        y_pred = model(x_true)                              # Forward
        optimizer.zero_grad()
        loss = loss_fn(y_pred, y_true)
        loss.backward()                                     # Backward
        steps_loss_list.append(loss.clone().detach())
        optimizer.step()
      train_loss = torch.tensor(steps_loss_list).mean()
      epoch_loss_list.append(train_loss)
      pbar.set_postfix(loss=f"{train_loss}")

      if epoch%validation_freq==0:
        val_step_loss_list = []
        val_correct_answers = 0
        val_set_size = 0
        with torch.no_grad():
          for x_val_true, y_val_true in val_dataloader:
            y_val_pred = model(x_val_true)

            loss_val = loss_fn(y_val_pred, y_val_true)
            val_step_loss_list.append(loss_val)

            correct_answers = torch.sum(y_val_true==torch.argmax(y_val_pred, dim=-1))
            val_correct_answers += correct_answers
            val_set_size += y_val_true.numel()

          val_loss = torch.tensor(val_step_loss_list).mean()
          val_loss_list.append(val_loss)
          val_acc_list.append(val_correct_answers/float(val_set_size))

  return model, epoch_loss_list, val_loss_list, val_acc_list

def get_test_acc(model, test_dataset, batch_size, device):

  if not str(device)=="cpu":
    test_dataset = CudaDataset(test_dataset, device)

  test_dataloader = DataLoader(test_dataset, batch_size=batch_size, shuffle=True)

  y_pred = []
  y_test = []
  val_step_loss_list = []
  test_correct_answers = 0
  test_set_size = 0

  for test_x_true, test_y_true in test_dataloader:
    y_test.append(test_y_true)
    y_test_pred = model(test_x_true)
    y_pred.append(y_test_pred.detach())

    correct_answers = torch.sum(test_y_true==torch.argmax(y_test_pred, dim=-1))
    test_correct_answers += correct_answers
    test_set_size += test_y_true.numel()

  test_acc = test_correct_answers/float(test_set_size)

  return test_acc

Como podemos ver el optimizador no est√° haciendo grandes avances ajustando los pesos de nuestro modelo. Tras 200 √©pocas, el error de entrenamiento ha bajado muy poco. Esto indica que o los pasos de ajuste que se est√°n produciendo son muy peque√±os (que podr√≠amos ajustar alterando el ratio de aprendizaje) o que el gradiente no es muy estable.


El gradiente determina la direcci√≥n de actualizaci√≥n de los pesos. Cuando el gradiente es inestable y cambia de forma brusca, el avance en la direcci√≥n "correcta" puede ser lento. Para compensarlo existe el momentum. La intuici√≥n f√≠sica del momentum puede ser el impulso (o la inercia) que lleva un objeto. La idea es que los gradientes pasados afecten a la actualizaci√≥n del momento actual.


Matem√°ticamente se puede describir como


$$\Delta g_{t} = \mu \Delta g_{t-1} +  \nabla L(m_{\mathbf{w}_{t}}(x))$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \Delta g_{t}$$


Con $\mu$ como el valor de momentum que aplicamos y $\Delta g_{t}$ como el incremento (o m√°s bien decremento) que se le aplica al gradiente en el tiempo $t$ (No confundir con el $\Delta$ que usamos en el primer cuaderno para nuestro primitivo m√©todo de ajuste). Esencialmente, estamos teniendo en cuenta el vector con el que incrementamos (o m√°s bien decrementamos) el gradiente en el paso anterior para a√±adir una parte de ese incremento (regulado por $\mu$) tambi√©n a este paso. $\mu$ suele tener valores entre 0.9 y 0.99 cuando se aplica, lo que es darle mucha importancia a los pasos anteriores para actualizar el gradiente. El momentum va a tener como efecto directo que los pasos de ajuste sean m√°s grandes y que crezcan si la direcci√≥n de actualizaci√≥n se mantiene. Se puede imaginar como una pelota rodando por una colina irregular. Cuanto m√°s rueda en una direcci√≥n, m√°s r√°pido rueda en esa direcci√≥n.


Adem√°s de poder ayudar a acelerar la convergencia, el momentum ayuda a evitar m√≠nimos locales porque aunque se llegue a un puntoen el que el gradiente sea muy bajo (o incluso 0), el factor acumulado de los gradientes anteriores va a provocar la actualizaci√≥n de los pesos. Esto se puede imaginar como si en la colina por la que tiramos nuestra pelota hubiera peque√±os agujeros. Si la pelota fuera despacio, una vez dentro de esos agujeros no podr√≠a salir, pero al llevar impulso por la ca√≠da es capaz de superar una pendiente de subida peque√±a.


M√°s informaci√≥n


En el SGD el momentum se puede establecer s√≠mplemente indic√°ndolo en el argumento correspondiente (link a la API).

Hay un segundo factor, el dampening ($\tau$) que permite modular la influencia del gradiente actual respecto al acumulado del momentum.


$$\Delta g_{t} = \mu \Delta g_{t-1} +  (1-\tau) \nabla L(m_{\mathbf{w}_{t}}(x))$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \Delta g_{t}$$


Sin embargo, no es un factor que suela usarse con tanta frecuencia.






















Si uno observa la informaci√≥n del SGD en Pytorch, observa que hay algunos argumentos m√°s configurables. Uno muy interesante es el weight decay o disminuci√≥n del peso (controlado con el factor $\lambda$). La disminuci√≥n de los pesos se a√±ade a la funci√≥n matem√°tica como sigue:


$$\Delta g_{t} = \mu \Delta g_{t-1} +  (1-\tau) (\nabla L(m_{\mathbf{w}_{t}}(x))+\lambda \mathbf{w}_{t})$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \Delta g_{t}$$


Este factor tiene un objetivo: que los pesos del modelo sean lo m√°s bajos posibles. Supongamos que no aplicamos momentum ni dampening ($\mu=0$ y $\tau=0$) y que el gradiente fuera nulo ($\nabla L(m_{\mathbf{w}_{t}}(x))=0$) para ver m√°s claro qu√© hace ese elemento de la ecuaci√≥n:


$$\Delta g_{t} = \lambda \mathbf{w}_{t}$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \Delta g_{t}$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \lambda \mathbf{w}_{t}$$


La actualizaci√≥n de los pesos va a acercar los pesos a 0 (en un caso extremo, con $\gamma=1$ y $\lambda=1$, los pesos pasar√≠an a 0 en un solo paso de actualizaci√≥n en este caso).


¬øPor qu√© es interesante mantener los pesos con valores bajos? En teor√≠a, contribuye a la generalizaci√≥n disminuyendo el sobreajuste. La intuici√≥n tras esto es que unos pesos muy grandes provocan que los cambios en las entradas del modelo tengan un efecto mucho mayor (recordemos que los pesos multiplican las entradas), mientras que unos pesos peque√±os minimiza el efecto de los cambios de la entrada. Dos entradas parecidas (cambios peque√±os en la entrada) queremos normalmente que provoquen un comportamiento similar en el modelo.


Esto es una espada de doble filo. Una disminuci√≥n del peso demasiado grande provoca el underfitting (infraajuste): impide que el modelo aprenda.


Este concepto de imponer restricciones condicionando c√≥mo debe ajustarse el modelo es lo que llamamos regularizaci√≥n.

El √∫ltimo elemento que podemos ver si inspeccionamos el SGD en la API de Pytorch es la Aceleraci√≥n de Nesterov. Este concepto es una alterntiva conceptualmente relacionada con el momentum.


$$\Delta g_{t} = \mu \Delta g_{t-1} + \gamma \nabla L(m_{\mathbf{w}_{t}-\mu \Delta g_{t-1} }(x))$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \Delta g_{t}$$


La propuesta es, durante la actualizaci√≥n de los pesos calcular el gradiente en la posici√≥n adelantada donde nos dejar√° el momentum ($\mu \Delta g_{t-1}$) y corregir la actualizaci√≥n usando ese gradiente "futuro". Esto te√≥ricamente facilita la convergencia al permitir "corregir" antes una direcci√≥n err√≥nea.


PyTorch no implementa esta formula tal cu√°l, sino una reordenaci√≥n algebraica del c√°lculo para hacerlo m√°s eficiente, pero te√≥ricamente equivalente.


$$\Delta g_{t} = \nabla L(m_{\mathbf{w}_{t}}(x)) + \mu(\mu \Delta g_{t-1} + \nabla L(m_{\mathbf{w}_{t}}(x)))$$


$$\mathbf{w}_{t+1} = \mathbf{w}_{t} - \gamma \Delta g_{t}$$


La verdad es que la aceleraci√≥n de Nestorov se usa poco.






















Alterar el Ratio de Aprendizaje¬∂
Otra t√©cnica que vale la pena comentar ahora es la manipulaci√≥n del ratio de aprendizaje durante el entrenamiento (normalmente para ir reduci√©ndolo). La intuici√≥n es sencilla: seg√∫n avanza el entrenamiento, nos puede interesar que nuestros pasos de actualizaci√≥n de pesos sean m√°s peque√±os porque deber√≠amos estar m√°s cerca de la soluci√≥n. Los pasos grandes pueden ser demasiado grandes y "pasar la soluci√≥n de largo" en lugar de acercarse a ella de manera consistente y estable. La idea suele ser avanzar m√°s despacio cuanto m√°s avanzamos.


Estas t√©cnicas en PyTorch (Y en Tensorflow) est√°n desligadas de los optimizadores y se gestionan con unas clases que heredan de torch.optim.LRScheduler. Un scheduler (planificador) es una clase que gestiona el avance del ratio de aprendizaje. Como es una clase nueva, necesitmaos introducirlo en nuestro bucle de entrenamiento para usarlo. Para ello simplemente contemplamos el paso del scheduler en cada √©poca.


















In¬†[29]:






def learning_loop_for_classification_with_scheduler(train_dataloader, val_dataloader, model, epochs, loss_fn, optimizer, scheduler, validation_freq):
  epoch_loss_list = []
  val_loss_list = []
  val_acc_list = []

  with tqdm(range(epochs), desc="epoch") as pbar:
    for epoch in pbar:
      steps_loss_list = []
      for x_true, y_true in train_dataloader:
        y_pred = model(x_true)                              # Forward
        optimizer.zero_grad()
        loss = loss_fn(y_pred, y_true)
        loss.backward()                                     # Backward
        steps_loss_list.append(loss.clone().detach())
        optimizer.step()
      train_loss = torch.tensor(steps_loss_list).mean()
      epoch_loss_list.append(train_loss)
      pbar.set_postfix(loss=f"{train_loss}")

      if epoch%validation_freq==0:
        val_step_loss_list = []
        val_correct_answers = 0
        val_set_size = 0
        with torch.no_grad():
          for x_val_true, y_val_true in val_dataloader:
            y_val_pred = model(x_val_true)

            loss_val = loss_fn(y_val_pred, y_val_true)
            val_step_loss_list.append(loss_val)

            correct_answers = torch.sum(y_val_true==torch.argmax(y_val_pred, dim=-1))
            val_correct_answers += correct_answers
            val_set_size += y_val_true.numel()

          val_loss = torch.tensor(val_step_loss_list).mean()
          val_loss_list.append(val_loss)
          val_acc_list.append(val_correct_answers/float(val_set_size))
    scheduler.step()

  return model, epoch_loss_list, val_loss_list, val_acc_list

Alternativas al SGD¬∂
Adagrad¬∂
La idea tras el AdaGrad (Adaptative Gradient) es adaptarse al gradiente de cada peso. Para el $i$-√©simo peso plantea la siguiente regla de actualizaci√≥n en el paso $t$:


$$v_{t,i}=v_{t-1,i} + (\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}})^2$$


$$w_{t+1,i} = w_{t,i}-\frac{\gamma}{\sqrt{v_{t}}+\epsilon}\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}}$$


Aunque la matem√°tica pueda parecer un poco compleja, la idea es sencilla: que cada peso tenga un ratio de aprendizaje en relaci√≥n al tama√±o de los gradientes anteriores. El cuadrado al calcular $v_{t,i}$ y la posterior raiz cuadrada es solo para asegurar que el valor sea positivo. A m√°s peque√±os sean los gradientes de un peso, mayor va a ser su ratio de aprendizaje. A m√°s grande, menor. Esto puede ir en contra de la intuici√≥n. Si un gradiente es mayor, ser√° que hay una inclinaci√≥n mayor respecto a la p√©rdida y, por tanto, hace falta que el peso se actualice m√°s, ¬øno? A veces s√≠, a veces no.


Imaginemos un peso que tiene muy poca relevancia la mayor√≠a del tiempo (con lo que su gradiente es muy peque√±o), pero hay algunos ejemplos (pocos) para los que es cr√≠tico (su gradiente es muy grande). Si ese peso solo se actualiza de manera relevante con esos ejemplos en los que es cr√≠tico, su avance hacia el m√≠nimo es muy despacio la mayor√≠a del tiempo con grandes saltos puntuales. Es muy inestable. Si le damos m√°s importancia a ese gradiente del que deber√≠a tener cuando es poco relevante, conseguimos que su avance sea m√°s constante. Por el contrario, si a la vez hay alg√∫n peso que es cr√≠tico casi siempre (su gradiente es muy grande), ralentizamos su aprendizaje para intentar nivelarlo con los pesos m√°s "lentos".


AdaGrad t√≠picamente funciona bien con conjuntos de datos conocidos como sparse o dispersos. Conjuntos de datos en los que muchas de las caracter√≠sticas no son relevantes la mayor√≠a del tiempo, pero puntualmente tienen mucha relevancia.


Esto por s√≠ mismo no evita los m√≠nimos locales, pero se puede combinar con el momentum.


Problema: el crecimiento de $v_{t,i}$ es mon√≥tono. Siempre va a m√°s, as√≠ que inevitablemente con el tiempo el ratio de aprendizaje de todos los pesos va a ir disminuyendo. AdaGrad va ralentizando el aprendizaje con el paso de las √©pocas y puede llegar a ser contraproducente.


Lo importante del AdaGrad es la idea de adaptar los ratios de aprendizaje.

RMSProp¬∂
RMSProp (Root Mean Square Propagation) sigue una idea parecida a la de AdaGrad, pero en lugar de ir acumulando los cuadrados sin m√°s, los promedia.


$$v_{t,i}=\beta v_{t-1,i} + (1-\beta)(\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}})^2$$


$$w_{t+1,i} = w_{t,i}-\frac{\gamma}{\sqrt{v_{t}}+\epsilon}\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}}$$


Simplemente a√±adiendo ese factor $\beta$ permite "olvidar" los gradientes pasados, con lo que el factor $v_{t,i}$ ya no tiene por qu√© crecer siempre. Soluciona el problema del AdaGrad.

Adam¬∂
Probablemente el optimizador m√°s usado. Adam (Adaptative Moment Estimation) combina las ideas de los optimizadores anteriores con el momentum y una correcci√≥n del sesgo. Aunque t√≠picamente en la literatura del Adam se habla de momentum de primer orden ($\beta_{1}$) y momentum de segundo orden ($\beta_{2}$). El momentum de primer orden es el concepto de momentum que hemos visto anteriormente y el de segundo orden, la adaptaci√≥n autom√°tica del ratio de aprendizaje que hemos visto en RMSProp. Por ser consistentes, para el momentum de primer orden usaremos la $\mu$ y la $\beta$ para el otro.


$$m_{t,i} = \mu m_{t-1,i} + (1-\mu)\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}}$$


$$\widehat{m_{t,i}} = \frac{m_{t,i}}{1-\mu^t}$$


$$v_{t,i}=\beta v_{t-1,i} + (1-\beta)(\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}})^2$$


$$\widehat{v_{t,i}} = \frac{v_{t,i}}{1-\beta^t}$$


$$w_{t+1,i} = w_{t,i}-\frac{\gamma\widehat{m_{t,i}}}{\sqrt{\widehat{v_{t}}}+\epsilon}$$






















Esencialmente estamos mezclando los dos elementos, pero hemos a√±adido unas operaciones que tienen como objetivo corregir el sesgo inicial de los componentes $m_{t,i}$ y $v_{t,i}$. Su utilidad es f√°cil de ver. En todos estos algoritmos esos valores que "acumulamos" comienzan a 0. Con t=1, $m_{t-1,i}=0$ y $v_{t-1,i}=0$. Si nos planteamos esa primera iteraci√≥n y tenemos un valor de $\mu$ alto (el art√≠culo original recomienda $\mu=0.9$), nuestro $m_{1,i}=0+0.1\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}}$. Lo mismo para un valor $\beta$ alto (en el art√≠culo original recomiendan $\beta=0.999$), $v_{1,i}=0 + 0.001(\frac{\partial L(m_{\mathbf{w}_{t}}(x))}{\partial w_{t,i}})^2$. Los valores al principio del entrenamiento de $m_{t,i}$ y $v_{t,i}$ ser√°n muy bajos para todos los pesos. Para compensarlo se incluye esa correcci√≥n. Los denominadores ser√°n muy bajos en los primeras iteraciones para aumentar el valor de $\widehat{m_{t,i}}$ y $\widehat{v_{t,i}}$, pero al avanzar $t$ pasar√°n a ser irrelevantes.


¬°Vamos a verlo en acci√≥n con sus par√°metros por defecto!

Conclusiones¬∂
Aunque la idea de utilizar el gradiente para ajustar los pesos de las redes neuronales es general, existen diversos criterios de ajuste posibles. Lo ideal ser√≠a conocer el camino perfecto para llegar al m√≠nimo absoluto esquivando m√≠nimos locales, pero como solo podemos hacer suposiciones, hay diversas maneras de intentar que ese camino sea lo m√°s estable y directo posible.


Tanto el momentum como el ajuste de la tasa de aprendizaje de manera adaptativa para cada peso son dos criterios bastante efectivos que combinados dan lugar al optimizador Adam.
